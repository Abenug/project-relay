# Project Relay

# Design Principles

Version: 1.0

---

# Purpose

Project Relay is designed to improve the existing package delivery process used by public transport drivers.

The application should feel simple, trustworthy, and effortless to use for drivers, senders, and receivers.

Every design decision should follow the principles below.

---

# 1. Solve Real Problems

Every feature must solve a real problem experienced by users.

If a feature does not reduce confusion, save time, or improve trust, it should not be included.

---

# 2. Show What Matters Now

The application should always prioritize information that requires the user's immediate attention.

Active deliveries should always appear before completed deliveries.

The Home screen should focus on "Right Now."

---



# 3. One Account, Multiple Capabilities

Users have one account.

A user may become a sender, receiver, or driver at different times without switching accounts.

The interface adapts automatically based on the user's current capabilities and activities.

---



# 4. Minimize User Effort

Reduce unnecessary typing and unnecessary taps.

Whenever possible, automate actions instead of asking users to remember them.

Examples:

- Automatic delivery updates.
- Automatic tracking after driver acceptance.
- Deep links from SMS.

---



# 5. Speak Like a Human

Use clear, friendly language instead of technical terms.

Examples:

Good:

- Right Now
- Your package is on its way.

Avoid:

- Activity Feed
- Status: In Transit

---



# 6. Build Trust

Users should always know:

- Where the package is.
- Who is transporting it.
- When it is expected to arrive.

The application should reduce uncertainty throughout the delivery process.

---



# 7. Design for Growth

The first version should remain simple while allowing future expansion.

Features added later should fit naturally into the existing experience without requiring major redesigns.

---



# 8. Keep Drivers Focused

Drivers should spend as little time as possible interacting with the application.

The interface should support quick actions and avoid unnecessary distractions.

---



# 9. Consistency

Similar actions should behave similarly throughout the application.

Navigation, terminology, and layouts should remain predictable and easy to learn.

---



# 10. Simplicity Over Complexity

When multiple solutions exist, prefer the simpler one if it provides a similar user experience.

The best feature is often the one users never need to think about.

# 11. Work With Existing Habits

Project Relay improves the existing package transportation process instead of replacing it.

The application should fit naturally into how people already send and receive packages through public transport.

Technology should support the process, not force users to adopt an entirely new one.
