# Project Relay

# Design Principles

Version: 1.0

---

# Purpose

Project Relay is designed to improve the existing package transportation process through public transport drivers.

The application should make package transportation more visible, trustworthy, and convenient while respecting how people already send and receive packages.

Every design decision should follow these principles.

---

# 1. Respect Existing Transportation Systems

Project Relay does not replace public transport drivers or create a delivery network.

Drivers already have their own journeys between locations.

The application only provides a digital system that helps them carry packages on routes they are already traveling.

Technology should support existing habits instead of forcing users to change their behavior completely.

---

# 2. Solve Real Problems

Every feature must solve a real user problem.

Features should focus on reducing:

- Uncertainty
- Repeated phone calls
- Communication problems
- Lack of package visibility

If a feature does not improve trust, save time, or reduce confusion, it should be reconsidered.

---

# 3. Show What Matters Right Now

The application should prioritize information that needs the user's attention at the current moment.

The Home screen focuses on:

"Right Now"

Active situations should appear before old history.

Users should quickly understand:

- What is happening?
- Where is my package?
- What action do I need to take?

---

# 4. One Account, Multiple Capabilities

Users should have one account.

The same person may become:

- Sender
- Receiver
- Driver

at different times.

The application should adapt to the user's activities instead of forcing multiple accounts.

---

# 5. Focus on Packages, Not Drivers

Users care about their package, not only the person transporting it.

The application should organize information around:

- Package status
- Package location
- Package progress

Driver information exists to create trust and transparency.

---

# 6. Minimize User Effort

The application should reduce unnecessary actions.

Whenever possible:

- Automate updates.
- Avoid repeated input.
- Use information already available.
- Remove unnecessary buttons.

Examples:

- Automatic tracking after trip movement begins.
- SMS links directly opening package tracking.
- Remembering logged-in users.

---

# 7. Keep Drivers Focused

Drivers are often busy and operating vehicles.

The application should require minimum interaction from drivers.

Design goals:

- Simple actions.
- Clear information.
- No unnecessary steps.
- Avoid distracting workflows.

The driver should focus on driving, not managing an app.

---

# 8. Build Trust Through Transparency

The main value of Project Relay is trust.

Users should always understand:

- Who is carrying the package.
- Where the package is.
- What route it is following.
- When it is expected to arrive.

The system should reduce uncertainty.

---

# 9. Speak Like a Human

The application should use simple and friendly language.

Avoid technical terms.

Examples:

Good:

"Your package is on the way."

"Driver is 20 minutes away."

"What's happening right now?"

Avoid:

"Status: In Transit"

"Location synchronization active"

"Delivery entity updated"

---

# 10. Design Around Real User Behavior

The application should match how people already work.

Current process:

- People find drivers at stations.
- Drivers share phone numbers.
- Receivers call for updates.

Project Relay should improve this process, not completely replace it.

---

# 11. Simplicity Over Complexity

When multiple solutions exist, prefer the simplest one.

A feature should only be added if its value is greater than the complexity it introduces.

Simple experiences are easier for everyone:

- Drivers
- Senders
- Receivers

---

# 12. Design for Growth

The first version should be simple but prepared for future expansion.

The system should allow future improvements such as:

- More routes
- More drivers
- More users
- Additional services

without requiring a complete redesign.

---

# 13. Consistency

Similar actions should behave similarly throughout the application.

Navigation, terminology, and user interactions should remain predictable.

Users should not need to learn the application repeatedly.

---

# Final Design Philosophy

Project Relay is not about creating a new delivery system.

It is about making an existing transportation process safer, clearer, and more reliable through technology.
